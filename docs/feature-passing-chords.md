# Feature Spec: Passing Chords

> **Status: IMPLEMENTED** ‚Äî This document is the original design spec. The authoritative description of the current implementation is in **`docs/architecture.md` ¬ß16**. The actual implementation diverges from this spec in several details (density formula, chain styles, snap grid). Use this doc for background/context only.

## Overview

Passing chords are short, transient chords inserted **between** structural chord changes to create smooth harmonic motion. They occupy rhythmically weak positions (non-beat slots) and lead into the next structural chord using common voice-leading techniques: secondary dominant, secondary diminished, tritone substitution, or diatonic step-approach.

This document is the authoritative specification for the implementation.

---

## 1. Background: rhythmDNA and Beat/Non-Beat Slots

### What rhythmDNA Is

`rhythmDNA` (called `rankedArray` in code) is a flat `Array<number|null>` encoding beat hierarchy across all slots of a musical phrase:

| Value | Meaning |
|---|---|
| `null` | No note at this slot (rest) |
| `0` | Highest-priority slot (downbeat) |
| `1, 2, ‚Ä¶` | Successively weaker positions |

Generated by:
1. `generateDeterministicRhythm()` in `src/generation/rhythmicPriorities.js` ‚Äî builds the base weight template per time signature
2. `generateRankedRhythm()` in `src/generation/generateRankedRhythm.js` ‚Äî flattens, applies `rhythmVariability`, returns the final ranked array

### Beat vs Non-Beat Slots

For passing chords, the relevant definition is **quarter-note beat position**:

```
beatLength = (TICKS_PER_WHOLE * timeSignature[0] / timeSignature[1]) / timeSignature[0]
           = TICKS_PER_WHOLE / timeSignature[1]
```

- **On a beat**: `offset % beatLength === 0` (exactly on a quarter-note pulse)
- **Off a beat**: everything else ‚Äî eighth-note upbeats, sixteenth notes, etc.

Examples for common time signatures (`TICKS_PER_WHOLE = 48`):

| Time sig | Measure length | Beat length | Beat positions (ticks) |
|---|---|---|---|
| 4/4 | 48 | 12 | 0, 12, 24, 36 |
| 3/4 | 36 | 12 | 0, 12, 24 |
| 6/8 | 36 | 6 | 0, 6, 12, 18, 24, 30 |
| 2/4 | 24 | 12 | 0, 12 |
| 5/4 | 60 | 12 | 0, 12, 24, 36, 48 |

### Current Chord Timing

Chord changes are generated via:
1. `generateDeterministicRhythm(1 measure, ts, 16ths resolution)` ‚Üí `globalTemplate`
2. `MelodyGenerator` with `notesPerMeasure = chordSettings.chordCount`, `randomizationRule: 'progression'`
3. Backward-planning distributes `Chord[]` onto active rhythm slots

At low `chordCount` values (1‚Äì2), all generated chord slots land on beat positions. At higher values (3‚Äì4), some slots may fall on off-beats depending on the rankedArray and rhythmVariability.

| Old options | New options |
|---|---|

This allows up to 4 structural chords per measure (one per quarter-note beat in 4/4), leaving room for passing chords at 8th-note off-beat positions.

**File:** `src/components/controls/rows/InstrumentRow.jsx` ‚Äî the `allowedValues` and `options` arrays in the `chordCount` stepper.

### 2.3 Passing Chord Architecture: Two-Phase Planning

The critical invariant is that **backward-planning for fixed progressions must only see structural (beat-aligned) slots**. If off-beat slots participate in backward planning, the progression pool gets redistributed across too many positions, scrambling Pachelbel, Andalusian, and similar cadences.

**Step 1 ‚Äî Structural chord generation (beat-aligned, `useMelodyState.js`)**

When `passingChords` is enabled, `rhythmVariability` is forced to `0` for the structural chord layer. This guarantees all structural chord slots land on beat positions (quarter-note multiples). Backward-planning distributes the fixed progression only over beat-aligned slots ‚Äî cadence integrity is preserved regardless of `chordCount`.

**Step 2 ‚Äî Passing chord insertion (`insertPassingChords`, `passingChords.js`)**

Called after structural generation. Structural chord Melodies have NO empty gaps: `offset[i] + duration[i] === offset[i+1]` always. Passing chords are inserted by subdividing the preceding structural chord's duration.

**Phase 1 ‚Äî Slot classification:**  
For each consecutive structural chord pair (A ‚Üí B), collect all 8th-note off-beat positions strictly inside A's duration. These are the candidate PASSING SLOTS.

**Phase 2 ‚Äî Chord assignment per gap:**

| Decision | Probability | Behavior |
|---|---|---|
| 0 passing chords | ~40% | Skip gap entirely |
| 1 passing chord | ~40% | Random approach type |
| N‚â•2 passing chords | ~20% | Sequential approach chain (see 2.4) |

The last N candidate positions in the gap are chosen so the chain arrives close to the next structural chord (most musical). Maximum passing chords per gap: 3 (`MAX_PASSING_PER_GAP`).

### 2.4 Sequential Approach Chains (N‚â•2 Passing Chords)

When 2 or more passing chords are inserted in a single gap, they form an **approach chain** ‚Äî each chord resolves into the next, ending at structural chord B.

The chain is **built backwards** from B so each step naturally leads forward:
```
chain[N-1] = generatePassingChord(scale, B,          type)
chain[N-2] = generatePassingChord(scale, chain[N-1], type)
‚Ä¶
chain[0]   = generatePassingChord(scale, chain[1],   type)
```

Two chain styles (50/50 random):

**Chromatic chain**: alternates `secondary-dominant` and `tritone-sub`. Creates a cycle-of-fifths descent or alternating-tritone-sub pattern (jazz/bebop approach).

**Diatonic chain**: all steps use `diatonic` type (adjacent scale degrees walking toward B). Smooth stepwise motion within the key.

**`src/generation/passingChords.js`** ‚Äî `buildApproachChain(scale, targetChord, count, style, complexity)`

### 2.5 Passing Chord Types (single-chord case)

#### Type 1: Secondary Dominant (`secondary-dominant`)
- Root pitch class: `(nextRoot + 7) % 12` ‚Äî perfect fifth above the next chord's root
- Structure: dominant 7th `[0, 4, 7, 10]` semitones from root
- e.g., next = Dm ‚Üí passing = A7

#### Type 2: Secondary Diminished (`secondary-dim`)
- Root pitch class: `(nextRoot - 1 + 12) % 12` ‚Äî half step below next chord root
- Structure: fully-diminished 7th `[0, 3, 6, 9]`
- e.g., next = C ‚Üí passing = B¬∞7

#### Type 3: Tritone Substitution (`tritone-sub`)
- Root pitch class: `(nextRoot + 1) % 12` ‚Äî half step above next chord root
- Structure: dominant 7th `[0, 4, 7, 10]`
- e.g., next = Dm (sec dom = A) ‚Üí tritone sub = E‚ô≠7

#### Type 4: Diatonic Approach (`diatonic`)
- Root: diatonic scale degree immediately above or below next chord's degree (50/50)
- Build with `generateChordOnDegree(scale, adjacentDegree, complexity)`
- e.g., next = IV (degree 4) ‚Üí diatonic approach from III (degree 3) or V (degree 5)

#### Building Chromatic Chords (Types 1‚Äì3)

Types 1‚Äì3 may produce roots outside the diatonic scale. Build them directly from pitch-class + interval pattern:

```
function buildChordFromRoot(rootPC, semitoneIntervals, allNotes, octaveRef):
  // Find the actual note with octave near octaveRef
  root = allNotes.find nearest note with pitchClass === rootPC near octaveRef
  notes = semitoneIntervals.map(i => allNotes[indexOf(root) + i])
  return Chord(root, quality, notes, label, root, suffix, roman='?', ...)
```

Roman numeral can be set to `'?'` or derived as `bII`, `V/x`, etc. for display. For passing chords, a simplified display (e.g., `A7`, `B¬∞`, `E‚ô≠7`) is acceptable ‚Äî see Section 2.5.

### 2.5 Display: Passing Chord Notation

Passing chords are displayed with two visual differences from structural chords:

#### Smaller font
- Normal chord label: `fontSize="26"`, superscript `fontSize="16"`
- Passing chord label: `fontSize="21"` (‚âà20% smaller), superscript `fontSize="13"`

#### Right-arrow above the label
- A `‚Üí` symbol rendered above the chord root at a **fixed Y position**, independent of superscript height
- Fixed baseline: `CHORD_ROOT_Y - CHORD_SUPER_DY - 10` ‚Äî always above where superscripts can reach
- Font size: `10px`, color: same as chord label

**SVG structure for a passing chord:**

```jsx
<g key={key} data-offset={absoluteOffset} ...>
  {/* Fixed-height arrow ‚Äî not affected by superscript dy */}
  <text
    x={xPos}
    y={CHORD_ROOT_Y - CHORD_SUPER_DY - 10}
    fontSize="10"
    fontFamily="sans-serif"
    fill={chordColor}
    style={{ userSelect: 'none', pointerEvents: 'none' }}
  >‚Üí</text>
  {/* Chord label ‚Äî 20% smaller */}
  <text
    x={xPos}
    y={CHORD_ROOT_Y}
    textAnchor="start"
    fontFamily="serif"
    fontSize="21"
    fontWeight="normal"
    fill={chordColor}
    style={{ userSelect: 'none', pointerEvents: 'none' }}
  >
    {displayRoot}
    {displaySuffix && <tspan fontSize="13" dy={-CHORD_SUPER_DY} dx="2">{displaySuffix}</tspan>}
  </text>
</g>
```

**Detection:** `chord.meta?.isPassing === true`

### 2.6 Melody Influence

Passing chords automatically influence melody generation because:
- They are inserted into the `chordMelody` Melody object (same `offsets`/`notes`/`displayNotes` arrays)
- `MelodyGenerator.generateMelody()` uses `getChordAt(tickOffset)` which iterates `chordEvents` sorted by offset
- Any note at a tick offset during a passing chord's duration will be drawn from the passing chord's tonality

**No additional code needed** for this behavior ‚Äî it emerges from the shared `getChordAt` lookup.

## 6. Invariants

1. **Structural chord slots are beat-aligned** ‚Äî when `passingChords` is enabled, `rhythmVariability` is forced to `0` for structural chord generation so backward-planning distributes the fixed progression exclusively over quarter-note-beat positions. This preserves cadence sequences (Pachelbel, Andalusian, ii-V-I, etc.) regardless of `chordCount`.
2. **Structural chords are never moved or replaced** ‚Äî passing chords only subdivide the preceding structural chord's duration; structural offsets are unchanged.
3. **Passing chords only occupy off-beat positions** ‚Äî `pos % beatLength !== 0` is always true for any inserted passing chord.
4. **At most `MAX_PASSING_PER_GAP` (3) passing chords per structural gap** ‚Äî prevents overcrowding.
5. **Sequential chains are built backwards** ‚Äî each chord in a chain is generated as a passing chord leading into the next, ensuring clean voice-leading direction throughout the run.
6. **Passing chords obey the same Melody shape** ‚Äî `.notes`, `.displayNotes`, `.offsets`, `.durations` all present, enabling audio playback and display without special-casing downstream.
7. **`isPassing` flag lives in `meta`** ‚Äî `chord.meta.isPassing = true` is the single detection point; all rendering checks this flag.
8. **Arrow at fixed Y** ‚Äî the `‚Üí` arrow is rendered at `CHORD_ROOT_Y - CHORD_SUPER_DY - 10`, a fixed coordinate never altered by superscript `dy` offsets.

## 7. Edge Cases

| Scenario | Handling |
|---|---|
| chordCount = 0.25 with passingChords | No gaps between structural chords short enough for off-beat slots ‚Üí no passing chords generated (algorithm finds no eligible positions) |
| Time signature 6/8 | beatLength = 6 ticks; eligible off-beat positions are at tick 3 (eighth triplet position) ‚Äî may produce no passing chords if gap is only one beat long |
| rhythmVariability > 0 | Some structural chords may already be at off-beat positions; `isOnBeat` check correctly identifies these as structural (they had a chord generated there) |
| Transposition | `Chord.transpose()` is called on all chords when the user changes tonic. Passing chords transpose identically to structural chords ‚Äî no special handling needed since `meta.isPassing` is preserved through `clone()` / `transpose()` |
| Pagination / sliceChordsByRange | `sliceChordsByRange` and `getChordsWithSlashes` both iterate `displayNotes` by offset ‚Äî passing chords appear at their correct positions automatically |
| inputTestState | Passing chords will appear in chord recognition tests. This is intentional ‚Äî they add harmonic complexity |
